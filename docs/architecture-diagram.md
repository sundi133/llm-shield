## 7. Component Responsibility Matrix

| Component | Responsibility | Persistence | Scaling |
|---|---|---|---|
| **Votal Shield Worker** | Run guardrails, orchestrate pipeline | Stateless | Horizontal (N replicas) |
| **Redis** | Tenant configs, API keys, cache | AOF + RDB on volume | Active-replica |
| **Elasticsearch** | Event storage, alerting, dashboards | Persistent cluster | Cluster scaling |
| **Audit DB (SQLite)** | Compliance queries | Volume-backed | Single writer |
| **llama.cpp server** | LLM inference for slow-tier guardrails | GPU memory | Multi-GPU |

## 8. Key Design Principles

1. **Platform-Enforced Policies** вҖ” Tenants cannot disable or weaken their assigned guardrails. All policy is stored server-side in Redis.

2. **Tenant Isolation** вҖ” Each request is scoped to a tenant via API key resolution. Cross-tenant data access is prevented by RBAC scope boundaries.

3. **Stateless Workers** вҖ” Shield workers hold no tenant state. All config is loaded from Redis (with 60s in-memory cache). Workers can be scaled horizontally.

4. **Persistent Storage** вҖ” Redis uses append-only persistence (AOF) on Docker volumes. All tenant configs survive container restarts.

5. **Observability by Default** вҖ” Every request generates structured events in Elasticsearch with ECS-compliant fields, enabling compliance reporting and SIEM integration.

6. **Runtime Updates** вҖ” Tenant guardrail policies can be updated via `/v1/admin/tenants` API without restarting workers. New configs propagate within 60 seconds (cache TTL).
